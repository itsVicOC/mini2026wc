- 产品名称：苏式生活馆 - 2026 世界杯赛事伴侣
- 端类型：微信小程序
- 小程序名称：苏式生活馆
- 小程序 AppID：`YOUR_MINIPROGRAM_APPID`
- 文档版本：v0.2
- 编写日期：2026-06-12
- 当前阶段：需求确认稿
- 目标用户：关注 2026 年 FIFA 世界杯的普通球迷、轻度观赛用户、赛程追踪用户

## 2. 背景与目标

2026 年 FIFA 世界杯由加拿大、墨西哥、美国联合举办，赛事扩军至 48 支球队，总比赛数为 104 场，赛程跨度长、参赛队伍多、淘汰赛结构更复杂。用户需要一个轻量、打开即用的小程序来快速了解赛事进度、近期赛程、比分、积分榜、淘汰赛路径和射手榜。

本产品通过服务端定时调用 football-data.org API，将世界杯相关数据同步到数据库。小程序端只读取数据库中的结构化数据，避免客户端直接访问第三方 API，提升稳定性、安全性和响应速度。

### 2.1 产品目标

- 提供世界杯整体进度概览：总场次、已完赛场次、未进行场次、进行中场次。
- 提供当天及未来 7 天比赛安排，包括国家名称、比赛时间、比分、比赛状态。
- 提供小组赛页，支持按小组查看排名、积分、胜平负、进失球和本组赛程。
- 提供淘汰赛赛事安排，支持按阶段筛选，展示阶段、对阵、比分和晋级关系。
- 提供射手榜，展示球员、国家队、进球数等核心信息。
- 形成稳定的数据同步和缓存机制，确保小程序展示数据来自数据库。

### 2.2 不做范围

- 不提供用户登录、收藏球队、评论互动等社区功能。
- 不提供视频直播、图文直播、赔率、彩票、竞猜功能。
- 不在小程序端直接调用 football-data.org API。
- 不做多赛事平台，当前仅聚焦 2026 年 FIFA 世界杯。
- 不承诺实时秒级比分刷新，首版目标为分钟级数据更新。
- 未开始比赛不提供详情页，仅展示基础信息和开赛订阅入口。

### 2.3 已确认产品决策

- 技术方案：采用独立后端服务 + MySQL 数据库。
- 部署环境：已有云服务器，并通过 1Panel 管理部署。
- 第三方数据套餐：football-data.org 免费等级。
- 时间展示：默认展示北京时间。
- 小程序名称：苏式生活馆。
- 小程序 AppID：`YOUR_MINIPROGRAM_APPID`。
- 支持未开始比赛订阅开赛提醒；已订阅且未发送的提醒支持取消；暂不做收藏球队。
- 进行中和已结束比赛支持进入详情页查看技术统计、事件和阵容等数据。
- 暂无品牌色、Logo 或固定视觉参考，设计可围绕足球和世界杯氛围自主延展。

## 3. 用户与场景

### 3.1 用户画像

- 普通球迷：想快速知道今天有哪些比赛、比分是多少、谁晋级。
- 轻度观赛用户：只在世界杯期间关注赛事，偏好清晰、低学习成本的信息结构。
- 深度球迷：关注小组积分、淘汰赛路径、射手榜变化。

### 3.2 典型场景

1. 用户早上打开小程序，查看今天和未来一周的比赛安排。
2. 比赛结束后，用户查看首页进度和比分是否更新。
3. 小组赛期间，用户进入小组赛页查看某个小组的出线形势和本组比赛安排。
4. 淘汰赛期间，用户查看 16 强、8 强、半决赛、决赛的对阵与赛果。
5. 用户查看射手榜，了解当前进球最多的球员。

## 4. 功能需求

### 4.1 首页：赛事概览与近期赛程

首页是小程序的默认入口，强调“一眼知道世界杯进行到哪了，以及接下来有什么比赛”。

#### 4.1.1 赛事进度卡片

展示字段：

- 总比赛场次：默认 104 场，以数据库赛事配置为准。
- 已完赛场次：状态为 `FINISHED` 的比赛数量。
- 未进行场次：状态为 `SCHEDULED`、`TIMED`、`POSTPONED` 等未完赛状态的比赛数量。
- 进行中场次：状态为 `IN_PLAY`、`PAUSED`、`LIVE` 的比赛数量。
- 进度百分比：已完赛场次 / 总比赛场次。
- 异常状态：延期、中断、取消等状态可在进度卡片下方单独提示，避免用户误以为赛事信息缺失。

交互要求：

- 首页顶部直接展示，不需要点击。
- 使用足球、奖杯、进度环或球场草坪视觉元素增强世界杯氛围。

#### 4.1.2 今日赛程

展示当天比赛列表，按开赛时间升序排列。

展示字段：

- 比赛时间，默认转换为北京时间展示。
- 主队国家名称与国旗。
- 客队国家名称与国旗。
- 比分：已开赛或已完赛展示比分，未开赛展示 `-`。
- 比赛状态：未开始、进行中、已结束、延期。
- 比赛阶段：小组赛、32 强、16 强、8 强、半决赛、三四名决赛、决赛。

空状态：

- 当天无比赛时，展示“今天暂无比赛”，并引导查看未来一周。

#### 4.1.3 未来一周赛程

展示从今天起未来 7 天内的比赛，按日期分组。

展示字段：

- 日期与星期。
- 每场比赛的时间、队伍、比分、状态、阶段。

交互要求：

- 默认展开全部日期。
- 比赛较多时支持纵向滚动。

### 4.2 赛程页

赛程页用于查看完整比赛安排。

#### 4.2.1 筛选与分组

支持视图：

- 全部比赛。
- 小组赛。
- 淘汰赛。
- 按日期。
- 按阶段。

筛选项：

- 比赛状态：全部、未开始、进行中、已结束。
- 小组：A-L，适用于小组赛。
- 阶段：小组赛、32 强、16 强、8 强、半决赛、三四名决赛、决赛。

#### 4.2.2 比赛列表

展示字段：

- 比赛编号。
- UTC 开赛时间与本地展示时间。
- 主客队国家名称、国旗。
- 比分。
- 点球比分，如适用。
- 状态。
- 小组或阶段。
- 比赛地点，如 API 提供。

进行中和已结束比赛可点击进入详情页。详情页展示比分进程、比赛信息、关键事件、技术统计和阵容信息；未开始比赛不进入详情页。

### 4.3 小组赛页

小组赛页用于展示小组赛阶段各小组排名，并在当前小组下方展示本组赛程。

#### 4.3.1 小组切换

- 支持 A-L 共 12 个小组切换。
- 默认展示 A 组。
- 可横向滑动切换小组标签。

#### 4.3.2 积分榜字段

每支球队展示：

- 排名。
- 国家名称与国旗。
- 场次。
- 胜。
- 平。
- 负。
- 进球。
- 失球。
- 净胜球。
- 积分。

点击球队栏位可进入球队详情页，查看本届世界杯表现、比赛战况、后续赛程和队内射手。

展示规则：

- 前两名使用明显晋级标识。
- 可能晋级名额可在数据和赛制确认后补充说明。
- 数据来源为数据库中的 standings 表，不由前端计算主排名。

#### 4.3.3 本组赛程

当前小组积分榜下方展示本组全部比赛卡片。

展示与交互要求：

- 比赛卡片展示北京时间、状态、双方国家名称与国旗、比分或订阅按钮。
- 进行中和已结束比赛可点击进入比赛详情页。
- 未开始比赛展示“订阅”入口；已订阅且未发送的提醒可点击取消订阅。
- 比赛卡片中的球队名称可点击进入球队详情页。
- 小组切换时同步切换本组积分与本组赛程。

### 4.4 淘汰赛页

淘汰赛页用于展示从 32 强开始的完整淘汰赛安排。

#### 4.4.1 阶段视图

支持横向阶段筛选：

- 全部。
- 32 强。
- 16 强。
- 8 强。
- 半决赛。
- 三四名决赛。
- 决赛。

展示字段：

- 阶段名称。
- 比赛时间。
- 对阵双方。
- 比分。
- 加时赛比分，如适用。
- 点球比分，如适用。
- 胜者或晋级队。
- 比赛状态。

交互要求：

- 支持按“全部、32 强、16 强、8 强、半决赛、三四名决赛、决赛”筛选。
- 进行中和已结束比赛可点击进入比赛详情页。
- 未开始比赛展示“订阅”入口；已订阅且未发送的提醒可点击取消订阅。
- 已确定球队名称可点击进入球队详情页；未确定球队保持待定展示，不显示可点击样式。

#### 4.4.2 视觉要求

- 首版采用按阶段分组的列表式 bracket，保证移动端可读性。
- 后续可升级为横向滑动淘汰赛树。

### 4.5 射手榜页

射手榜展示当前赛事进球球员排名。

展示字段：

- 排名。
- 球员姓名。
- 所属国家队。
- 国家队旗帜。
- 进球数。
- 助攻数，如 API 提供。
- 点球数，如 API 提供。
- 出场数，如 API 提供。

排序规则：

- 默认按进球数降序。
- 进球数相同的情况下，若 API 已返回排名则遵循 API；否则按助攻数、出场数、球员名称排序作为本地展示兜底。

风险说明：

- football-data.org 的射手榜和部分详细统计字段可能受套餐权限限制。首版需在接入 API token 后验证 `/v4/competitions/WC/scorers` 对 2026 世界杯的可用性与字段完整度。
- 如套餐不支持射手榜，需确认是否升级套餐，或将射手榜降级为后续版本功能。

## 5. 信息架构

建议底部 Tab：

1. 首页：赛事进度、今日赛程、未来 7 天。
2. 赛程：完整比赛列表与筛选。
3. 小组赛：小组积分与本组赛程。

如需控制首版复杂度，可将“淘汰赛”和“射手榜”放入首页二级入口，但推荐保留独立 Tab，方便世界杯期间高频访问。

## 6. 数据来源与同步设计

### 6.1 第三方 API

数据源：football-data.org API v4

当前套餐：免费等级

核心接口规划：

- 比赛列表：`GET /v4/competitions/WC/matches`
- 单场比赛详情：`GET /v4/matches/{id}`
- 积分榜：`GET /v4/competitions/WC/standings`
- 射手榜：`GET /v4/competitions/WC/scorers`
- 队伍信息：`GET /v4/competitions/WC/teams`，如套餐和赛事阶段支持

认证方式：

- 服务端请求头携带 `X-Auth-Token`。
- API Token 不得出现在小程序端代码或前端网络请求中。

### 6.2 数据同步策略

服务端通过定时任务同步数据到数据库。

建议同步频率：

- matches：每 1 分钟同步一次，保障比分、状态和赛程变化的实时性。
- match_details：每 1 分钟同步一次，只同步正在进行和近期已结束的比赛详情。
- standings：每 5 分钟同步一次，更新小组积分。
- scorers：每 5 分钟同步一次，更新射手榜。
- teams：每天同步一次，更新球队名称、简称、队徽等低频变化信息。
- 手动同步：管理员可通过后端管理接口触发一次全量同步。

频控要求：

- 当前 football-data.org 免费等级额度为每分钟 10 次请求，默认同步策略需优先保障实时性，同时避免打满额度。
- 所有同步任务需记录执行日志、耗时、成功失败状态和错误信息。
- 当 API 请求失败时，前端继续展示数据库中的最近成功数据，并显示最后更新时间。
- 首页接口建议使用 30 秒左右短时缓存，减少数据库和后端服务压力，同时兼顾比分新鲜度。

### 6.3 数据流

1. 定时任务触发。
2. 服务端调用 football-data.org API。
3. 数据清洗、字段归一化、状态映射。
4. 通过 upsert 写入数据库。
5. 小程序通过自有后端接口读取数据库。
6. 小程序页面渲染赛事信息。

### 6.4 数据状态映射

football-data.org 状态需映射为小程序展示状态：

- `SCHEDULED` / `TIMED`：未开始。
- `IN_PLAY` / `LIVE`：进行中。
- `PAUSED`：中场或暂停。
- `FINISHED`：已结束。
- `POSTPONED`：延期。
- `SUSPENDED`：中断。
- `CANCELLED`：取消。

## 7. 数据库设计草案

数据库采用 MySQL，由独立后端服务读写。服务部署在已有云服务器上，并通过 1Panel 进行应用、数据库、环境变量和定时任务管理。

### 7.1 collections / tables

#### `competitions`

- `id`：内部 ID。
- `code`：赛事代码，如 `WC`。
- `name`：赛事名称。
- `season`：赛季年份，如 `2026`。
- `total_matches`：总场次，默认 104。
- `start_date`：开始日期。
- `end_date`：结束日期。
- `last_synced_at`：最后同步时间。

#### `teams`

- `id`：内部 ID。
- `api_team_id`：football-data.org team id。
- `name`：国家名称。
- `short_name`：简称。
- `tla`：三字母缩写。
- `crest`：队徽或国旗 URL。
- `group_name`：小组名，如 `GROUP_A`。
- `updated_at`：更新时间。

#### `matches`

- `id`：内部 ID。
- `api_match_id`：football-data.org match id，唯一索引。
- `competition_code`：赛事代码。
- `season`：赛季。
- `matchday`：比赛日。
- `stage`：阶段。
- `group_name`：小组名。
- `utc_date`：UTC 开赛时间。
- `local_date`：本地日期，便于查询。
- `status`：原始状态。
- `display_status`：展示状态。
- `home_team_id`：主队 ID。
- `home_team_name`：主队名称快照。
- `home_team_crest`：主队旗帜快照。
- `away_team_id`：客队 ID。
- `away_team_name`：客队名称快照。
- `away_team_crest`：客队旗帜快照。
- `home_score`：主队常规比分或全场比分。
- `away_score`：客队常规比分或全场比分。

- `extra_time_away_score`：加时客队比分。

- `winner_team_id`：胜者 ID。

- `referees_data`：裁判信息 JSON 文本。

- `competition_code`：赛事代码。

- `group_name`：小组名。

- `id`：内部 ID。

#### `sync_logs`

- `id`：内部 ID。

## 8. 后端接口设计草案

小程序端只访问独立后端服务提供的自有接口。

### 8.1 首页数据

`GET /api/home`

返回：

- 赛事进度。
- 今日比赛。
- 未来 7 天比赛。
- 最后同步时间。

### 8.2 赛程列表

`GET /api/matches`

查询参数：

- `date_from`
- `date_to`
- `stage`
- `group`
- `status`

### 8.3 比赛详情

`GET /api/matches/:apiMatchId`

仅用于进行中和已结束比赛，返回基础比赛信息和已同步的详情数据。

### 8.4 小组赛数据

`GET /api/standings`

查询参数：

- `group`

当前小组赛程复用 `GET /api/matches?group=GROUP_A`，不新增后端接口。

### 8.5 淘汰赛

`GET /api/knockouts`

返回按阶段分组的淘汰赛比赛。

### 8.6 射手榜

`GET /api/scorers`

查询参数：

- `limit`，默认 20。

## 9. 技术架构约束

### 9.1 部署架构

- 小程序端：微信小程序，负责页面展示和用户交互。
- 后端服务：独立 Web API 服务，负责读取 MySQL、提供小程序接口、保护第三方 API Token。
- 数据库：MySQL，存储比赛、球队、积分榜、射手榜、同步日志等数据。
- 定时任务：由后端服务或服务器计划任务触发，定时调用 football-data.org API 并写入 MySQL。
- 部署管理：通过 1Panel 管理后端服务、MySQL、环境变量、日志和任务。

### 9.2 环境变量

后端服务至少需要以下环境变量：

- `FOOTBALL_DATA_API_TOKEN`：football-data.org API Token。
- `MATCH_DETAILS_SYNC_CRON`：比赛详情同步频率。
- `MATCH_DETAILS_SYNC_LIMIT`：每轮比赛详情最多同步场次。
- `MYSQL_HOST`：MySQL 地址。
- `MYSQL_PORT`：MySQL 端口。
- `MYSQL_DATABASE`：数据库名。
- `MYSQL_USER`：数据库用户。
- `MYSQL_PASSWORD`：数据库密码。
- `APP_TIMEZONE`：默认 `Asia/Shanghai`。

### 9.3 安全约束

- API Token 只允许存在于服务器环境变量或 1Panel 密钥配置中。
- 小程序请求后端接口时不得携带 football-data.org Token。
- 后端接口需做基础限流，避免比赛日被高频刷新拖垮。
- 管理类同步接口如需暴露，必须增加鉴权，不允许公开访问。

## 10. 页面设计要求

### 10.1 设计关键词

- 足球感。
- 世界杯氛围。
- 大方、美观、清晰。
- 信息密度适中。
- 移动端优先。

### 10.2 视觉方向

- 色彩：深绿色草坪基调，搭配白色、金色、红色或蓝色作为强调色，避免过度花哨。
- 背景：可使用球场纹理、草坪线条、弧线、中圈、球网等轻量图形元素。
- 卡片：使用适度圆角和阴影，重点信息突出，避免过重装饰。
- 图标：足球、奖杯、哨子、日历、积分榜、球鞋等语义图标。
- 国旗 / 队徽：优先使用 API 返回的 crest 或后端维护的国家旗帜资源。

### 10.3 首页布局建议

1. 顶部：赛事标题、当前阶段、最后更新时间。
2. 进度区：总场次、已完赛、未进行、进行中。
3. 今日比赛：卡片列表。
4. 未来 7 天：按日期分组列表。

### 10.4 空状态与异常状态

- 无比赛：展示简洁文案，不使用错误样式。
- 数据同步失败：继续展示旧数据，并显示“数据更新可能延迟”。
- API 字段缺失：展示 `待定` 或 `-`，避免页面崩溃。

## 11. 非功能需求

### 11.1 性能

- 首页接口目标响应时间小于 500ms，不含冷启动。
- 小程序首页首屏应尽量少于 2 秒可见。
- 列表使用分页或按日期懒加载，避免一次渲染过多节点。

### 11.2 稳定性

- 数据同步任务失败不能影响小程序读取历史数据。
- API token、数据库密钥不得暴露在小程序端。
- 关键接口需有错误日志。

### 11.3 时区

- football-data.org 返回时间通常为 UTC。
- 数据库存储保留 UTC 时间。
- 前端默认转换为北京时间展示，并在页面标注。
- 查询今日和未来一周时需明确使用展示时区，避免跨日比赛错位。

### 11.4 合规与版权

- football-data.org 数据使用需遵守其套餐、署名和频控要求。
- 如使用 FIFA、世界杯官方标志、吉祥物、奖杯图片等视觉素材，需确认授权；首版应避免直接使用受保护官方商标图形。
- 小程序名称、图标、页面设计应避免让用户误认为官方 FIFA 产品。

首版建议埋点：

## 13. 验收标准

### 13.1 功能验收

- 首页能展示总场次、已完赛、未进行、进行中场次。
- 首页能展示当天比赛与未来 7 天比赛。
- 比赛比分在数据库更新后能正确展示。
- 小组赛页能按小组展示积分榜，并展示当前小组赛程。
- 淘汰赛页能按阶段筛选并展示比赛。
- 射手榜页能展示球员进球排名，或在 API 权限不足时有明确降级状态。
- 小程序端所有展示数据均来自数据库或自有接口，不直接访问 football-data.org。

### 13.2 数据验收

- 定时任务能成功拉取 matches 并 upsert 到数据库。
- standings、scorers 同步任务能记录成功或失败日志。
- 同步失败时不清空旧数据。
- 比赛状态、比分、队伍名称、阶段信息映射准确。

### 13.3 体验验收

- 首页首屏信息清晰。
- 页面风格具有足球和世界杯氛围。
- 无数据、加载中、失败状态都有可理解的展示。
- 主要页面在常见手机宽度下无内容重叠。

## 15. 已确认事项与待验证风险

### 15.1 已确认事项

1. 技术方案采用独立后端服务 + MySQL 数据库。
2. 服务部署在已有云服务器，并通过 1Panel 管理。
3. football-data.org API Token 使用免费等级。
4. 页面时间默认展示北京时间。
5. 小程序名称为苏式生活馆。
6. 小程序 AppID 为 `YOUR_MINIPROGRAM_APPID`。
7. 进行中和已结束比赛加入比赛详情页。
8. 暂不做收藏球队。
9. 暂无品牌色、Logo 或固定视觉参考。

### 15.2 待验证风险

1. football-data.org 免费等级是否支持 2026 世界杯 `matches`、`standings`、`scorers`、`teams` 接口。
2. 射手榜接口在免费等级下是否返回助攻、点球、出场数等扩展字段。

## 16. 主要风险

- 2026 世界杯赛程、球队、分组、赛果数据会随赛事进程持续变化，需依赖稳定的数据同步机制。
- football-data.org 免费或低阶套餐可能限制世界杯部分数据或射手榜字段。
- 第三方 API 数据延迟会直接影响比分与榜单新鲜度。
- 小程序审核可能关注体育赛事数据、名称、图标和版权描述，需避免官方误导。
- 比赛日访问峰值会显著增加，需要提前考虑缓存和接口限流。
